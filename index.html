Confidence · Methodology · Limitations

This tool produces estimates with approximately 90–95% confidence for well-formed input.

The score is regex-based code analysis plus a structured checklist. Confidence drops sharply when SQL is malformed, when the checklist is partially filled, or when the underlying platform includes features outside the rule set (Sybase IQ specifics, custom extended stored procedures, Replication Server topology, undocumented thick-client dependencies). Results are a triage aid, not a substitute for on-site review, interviews with the engineering team, or reading the actual code.

What the confidence band means

For a well-structured T-SQL file with clear CREATE boundaries and a substantially-filled checklist, the three scores (Code Complexity, Migration Readiness, Composite) will usually land within about one bucket of the reality that a human architect would report after on-site review. Confidence is lower when inputs are partial; the tool explicitly marks scores as Not assessed when there is no input for them. Numbers are never shown without a conf ~92% label.

How the three scores are computed

  • Code Complexity (0–100): derived from per-object rule matches and LOC tiers from the analyzer. Normalized across objects. Tiers: 0–30 LOW, 31–55 MEDIUM, 56–80 HIGH, 81–100 VERY HIGH.
  • Migration Readiness (0–100, higher = more gap): sum of weights of unchecked checklist items, normalized to percentage. Tiers: 0–25 READY, 26–50 MOSTLY READY, 51–75 MATERIAL GAPS, 76–100 SEVERE GAPS.
  • Composite Risk (0–100): 0.45 × CodeComplexity + 0.55 × Readiness. Readiness is weighted slightly heavier because human/process risks are harder to recover from than code. If only one input exists, the composite equals that one (with a caveat).
  • Effort estimate (person-months): calibrated to the playbook example (237 procs ≈ 9–14 PM) and inflated by the readiness gap. Range shown ±15% around the central estimate.

Known limitations

  1. This tool uses regex-based pattern matching, not a real SQL parser. Results are a triage aid, not a substitute for manual review.
  2. Scoring is rubric-based. Two objects with identical scores may take very different times to rewrite in practice.
  3. Business logic complexity is not captured — the tool only analyzes syntax, not semantics.
  4. No data leaves your browser. Nothing is uploaded, logged, or stored.
  5. Rewrite suggestions (if shown) are illustrative only and are NOT production-ready.
  6. Checklist answers are self-reported, not independently verified by the tool.
  7. Scoring weights are opinionated — based on the playbook methodology. Other engagements may warrant different weights.
  8. The effort formula is calibrated to one reference engagement and will vary significantly based on team skill, tooling maturity, and scope decisions.
