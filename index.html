Getting started

What is this tool for?
It operationalizes a Sybase→PostgreSQL / SQL Server / Oracle migration evaluation. You paste DDL, answer a structured readiness checklist, and the tool produces three live risk scores, a code-complexity heat map, a person-day effort breakdown, and a set of HTML / Markdown / JSON / slide-deck exports suitable for handing to an engineering team or an executive sponsor.
Do I need to install anything?
No. The entire tool is a single index.html file. Open it directly in a browser — or use the hosted GitHub Pages copy. There is no backend, no database, and no package to install.
Does any of my data leave the browser?
No. There are zero fetch() calls, zero CDN references, and zero external resources. The tool does not use localStorage or sessionStorage either. Your client's SQL and checklist answers never leave the tab. Closing the tab clears everything — if you need to resume a session, export the Raw JSON from the Reports tab and re-import (planned).
What's the fastest way to get a report?
  1. Paste Sybase T-SQL on the Analysis tab and click Analyze.
  2. Answer whatever you can on the Checklist tab. You don't need 100%.
  3. Go to the Reports tab and click Technical Report or Executive Summary.
You'll get a fully branded HTML report in under a minute. For an analysis-only or checklist-only run, you can skip either of the first two steps — the report will just mark the missing side as "Not assessed".

Analysis & rules

What does the analyzer actually detect?
60+ regex patterns covering Sybase-specific functions (ISNULL, GETDATE, CHARINDEX…), globals (@@identity, @@rowcount…), dynamic SQL (EXEC(), sp_executesql), cross-database refs (db..table), temp tables (#temp, ##temp), locking hints (NOLOCK, HOLDLOCK, READPAST…), cursors, GOTO, RAISERROR/PRINT, data types (MONEY, TEXT/IMAGE, TINYINT, TIMESTAMP row-version), legacy syntax (TOP, SELECT INTO #temp, pre-ANSI *= joins, COMPUTE, SET ROWCOUNT, USE <db>), identity, trigger pseudo-tables, transaction control, and deprecated DDL (sp_addtype, CREATE DEFAULT, CREATE RULE). See the full catalog on the About tab.
Why regex instead of a real parser?
Regex is cheap, fast, embedded inline in the HTML, and good enough for triage. A real parser (ANTLR, tree-sitter) would need a build step, a larger bundle, and would still need dialect-specific grammars. For the triage-and-scope use case, regex gets you 90% of the way for 5% of the cost. The tradeoff: false positives on unusual formatting, and false negatives on constructs outside the rule set.
Can I use this for non-Sybase source code?
Partially. The rule set is tuned for Sybase T-SQL, so rule matches will be noisy on Oracle / MySQL / DB2 / Informix sources. But the checklist, SQL Library, scoring model, reports, and slide decks are dialect-neutral and reusable. If that's your situation, skip the Analysis tab and work the Checklist — you'll still get a Migration Readiness score, a Composite Risk, an effort breakdown, and both report exports.

Target dialect (PostgreSQL / SQL Server / Oracle)

How do I switch migration targets?
On the Analysis tab, use the Target dialect segmented control (PostgreSQL / SQL Server / Oracle). The heat-map tooltips, per-object detail blocks, rule catalog, and Reports tab re-render in place. You don't need to re-analyze.
Which target is easiest to migrate to from Sybase?
SQL Server is the closest cousin — shares the same T-SQL ancestry, so many constructs carry over unchanged (GETDATE, ISNULL, @@rowcount, TRY/CATCH). PostgreSQL requires the most rewrite volume but has the richest modern feature set and is license-friendly. Oracle is the most distinctive — PL/SQL semantics, sequence+trigger patterns for identity, savepoint-based nested transactions, and a different exception model. The Composite Risk score is the same regardless of target; what changes is the per-rule rewrite cost.
Are the rewrite hints production-ready?
No. They're starting points. Every real rewrite needs human judgement about data types, NULL handling, transaction semantics, error codes, and performance characteristics. Use the hints to scope and estimate — not to paste directly into production.
What about MySQL / DB2 / Snowflake / BigQuery targets?
Not in the current rule set. The scoring model, checklist, and report structure are all dialect-neutral so the tool is still useful for those projects, but per-rule rewrite hints are Sybase→{PG, MSSQL, Oracle} only. If you'd like a rule pack for another target, email darshan.meel@gmail.com.

Scores & confidence

How is Code Complexity computed?
Each detected pattern has a score (0 for ANSI-portable rules, up to 5 for dynamic SQL). Per-object scores are summed and bucketed into Low / Medium / High / Very High. The 0–100 Code Complexity score is a normalised average of object scores, sized to the LOC footprint. Tiers: 0–30 LOW, 31–55 MEDIUM, 56–80 HIGH, 81–100 VERY HIGH.
How is Migration Readiness computed?
Sum the riskWeight (1–5) of every unchecked checklist item, divide by the total possible weight, multiply by 100. Higher = more gap. Tiers: 0–25 READY, 26–50 MOSTLY READY, 51–75 MATERIAL GAPS, 76–100 SEVERE GAPS. Unfilled = "Not assessed" — it does not default to zero, so an empty checklist does not flatter the score.
What is the Composite Risk?
0.45 × CodeComplexity + 0.55 × Readiness. Readiness is weighted slightly higher because human and process risks are harder to recover from than code. If only one input is available, the Composite equals that input with a caveat.
Where does the confidence number come from?
It is computed dynamically, not hardcoded. Starts at 55% baseline. +20% if the analyzer has run, +5% if the analyzed SQL actually contains Sybase-specific patterns (a dialect sanity check), -15% if it has been analyzed but no Sybase patterns were found (likely wrong dialect pasted). Up to +20% scaled by how much of the checklist you've answered. Floor 40%, ceiling 95%. The number in the header is live.
What's a "red flag" item?
Checklist items with redFlagIfUnchecked: true are the ones that would most likely cause the project to slip or fail if left unresolved — missing backups, no original architect, no source control, untested rollback procedure. They show a red marker next to the question and surface in the Top Delay Factors section of every report regardless of their raw weight.

Effort model

How are effort numbers calculated?
Effort is split into three workstreams — Code, Data, Infra — plus App which is flagged out-of-scope. Code uses per-object bucket days (Low 0.15 / Medium 0.5 / High 1.5 / Very High 3 person-days) plus small bumps for dynamic SQL and cursor usage. Data starts at 2 person-days and scales mildly with object count. Infra starts at 4 person-days with a premium for larger platforms. All three are then multiplied by a readiness inflation factor (up to ~40% extra for a SEVERE GAPS readiness score).
Why is App migration out of scope?
Application changes — connection strings, driver swaps, ORM configuration, endpoint updates — are almost always owned by a different team (usually the application engineering team) and require knowledge the database migration team doesn't have. Pretending to estimate them produces misleading totals. The report flags app migration explicitly as OUT OF SCOPE so the reader knows to get a separate estimate.
Do the per-object days add up to the total?
The per-object heat-map days equal the raw code-migration sum. The Reports tab Code Migration card also adds dynamic-SQL / cursor bumps and applies the readiness inflation multiplier, so its number is typically slightly higher than the heat-map column sum. Both come from the same computeEffortBand() function.
How do I adjust for my team's actual throughput?
Treat the number as the tool's calibrated baseline for a mid-level team. If your team is very senior, scale down 20–30%. If new to PL/pgSQL, scale up 20–40%. Always layer 20–30% contingency for UAT and go-live stabilisation on top — the tool does not include this.

Reports & exports

Which report do I send to whom?
Technical Report → the engineering team or DBA. Full analyzer findings, per-object heat map, rule catalog, and phase-by-phase checklist answers with notes. Executive Summary → the sponsor or decision-maker. 2–3 pages with three scores, top-3 macro risks, effort band by workstream, and an approach recommendation. Slide Deck → to present live on a call. Instagram-style carousel, keyboard nav. Methodology Deck → to the client at the start of the engagement, before any analysis has been run.
How do I export to PDF?
Open any HTML export in a browser and use Print → Save as PDF. The exports have a dedicated print stylesheet: the SAP-blue backdrop flattens to solid dark, margins tighten, and the content fills the page edge-to-edge.
What's in the Raw JSON?
The full state dump: meta (client, analyst, dates), scores, codeMetrics, all checklist answers with notes, computed delay factors, and effort band. Intended for archival or re-importing into a different tool. Re-import into this tool is on the roadmap.

Privacy & limitations

Where does the SQL go after I paste it?
Into a JavaScript variable in your browser tab. Nowhere else. No fetch, no XMLHttpRequest, no analytics, no telemetry. DevTools → Network → reload will show zero external requests.
Does refreshing the page lose my work?
Yes. There is deliberately no persistence. For multi-session engagements: export Raw JSON at the end of each session. Full re-import support is planned.
When should I NOT trust the output?
  • Malformed or minified SQL — regex can't recover from it
  • Source that isn't actually Sybase T-SQL — the analyzer will still match generic patterns but rewrite hints may be misleading
  • Features outside the rule set — custom extended stored procs, Sybase IQ columnar analytics, Replication Server topology, undocumented thick-client dependencies
  • A mostly-empty checklist — the readiness score requires self-reported data to mean anything
  • Any situation where a stakeholder is treating the report as a substitute for an on-site DBA review. It isn't.
How should I use the output in a conversation with a DBA?
As a scoping aid. "The analyzer flags ~40 Sybase-specific calls in these 7 procedures, 2 of which use dynamic SQL and 1 uses cursors. What's your take on whether that's representative of the rest of the platform?" is a much better opener than "how complex is your platform?" The tool gives you a concrete starting point for the real conversation.

What this tool can do for you

Evaluate a Sybase migration in a day, not a month.

This tool turns a pile of Sybase stored procedures and a series of discovery calls into three clear numbers, two written reports, and a pitch deck — all offline, all in a single HTML file. It is built for migration projects, whether you are evaluating an internal system or a platform owned by another organization. Use it to size the migration, spot the biggest risks before they become blockers, and communicate the findings to both engineers and decision-makers.

Pre-engagement materials

Download the methodology slide deck to show clients before any analysis starts. It explains what the evaluation covers, what inputs are needed, and what each phase delivers. It uses no client data — you can share it immediately.

Who this tool is for

  • DBAs and engineering leads planning a Sybase-to-PostgreSQL migration — estimate effort, find the hardest objects first, share a technical report with the team.
  • Consultants / evaluators running a migration-readiness review — structured checklist across six phases, auto-generated executive summary for sponsors.
  • Architects doing due-diligence or scoping — quickly see dynamic SQL, cross-DB joins, cursor usage, and other migration blockers.
  • Project managers who need an effort band and a list of delay factors to build a realistic plan.

What this tool does

  • Analyzes pasted Sybase T-SQL against 60+ regex patterns and surfaces rewrite hints for three migration targets: PostgreSQL, Microsoft SQL Server, and Oracle. Switch targets from the Analysis tab.
  • Tracks a 70-item migration-readiness checklist across six phases. Each item has a weight and a notes field.
  • Produces three live scores — Code Complexity, Migration Readiness, Composite Risk — with transparent formulas.
  • Generates a pre-engagement methodology deck, a technical report, an executive summary, a filled slide deck, and a raw JSON dump.
  • Keeps a copy-ready SQL discovery library for platform, inventory, connections, performance, and schema-quality queries.
  • Runs entirely in the browser — no network, no storage, no telemetry.

Other databases

The analyzer's 60+ regex rules are tuned for Sybase T-SQL, but rewrite hints are provided for three targets: PostgreSQL, Microsoft SQL Server, and Oracle. Pick the target dialect from the Analysis tab — heat map tooltips, detail blocks, and exported reports all update to show the chosen target's guidance. The checklist, SQL Library, scoring model, reports, and slide decks are dialect-neutral and translate to any relational source (MySQL, DB2, Informix, etc.) — only the per-rule rewrite strings are Sybase-sourced.

What the confidence band means

For a well-structured T-SQL file with clear CREATE boundaries and a substantially-filled checklist, the three scores (Code Complexity, Migration Readiness, Composite) will usually land within about one bucket of the reality that a human architect would report after on-site review. Confidence is lower when inputs are partial; the tool explicitly marks scores as Not assessed when there is no input for them. Numbers are never shown without a conf ~92% label.

How the three scores are computed

  • Code Complexity (0–100): derived from per-object rule matches and LOC tiers from the analyzer. Normalized across objects. Tiers: 0–30 LOW, 31–55 MEDIUM, 56–80 HIGH, 81–100 VERY HIGH.
  • Migration Readiness (0–100, higher = more gap): sum of weights of unchecked checklist items, normalized to percentage. Tiers: 0–25 READY, 26–50 MOSTLY READY, 51–75 MATERIAL GAPS, 76–100 SEVERE GAPS.
  • Composite Risk (0–100): 0.45 × CodeComplexity + 0.55 × Readiness. Readiness is weighted slightly heavier because human/process risks are harder to recover from than code. If only one input exists, the composite equals that one (with a caveat).
  • Effort estimate (person-months): baseline of 1 PM per ~8,000 LOC, plus complexity bumps for high/very-high objects and for dynamic SQL and cursor usage. The readiness gap inflates the estimate by up to 50%. Shown as a ±15% band. Actual effort varies significantly with team skill, tooling maturity, and scope.

The effort estimate is an order-of-magnitude starting point. Actual effort will vary based on team skill, tooling maturity, and scope decisions — treat the range as a planning starting point, not a commitment.

Limitations of this tool

This page describes exactly what this tool can and cannot do. Read it before sharing any report with a stakeholder.

Known limitations

  1. This tool uses regex-based pattern matching, not a real SQL parser. Results are a triage aid, not a substitute for manual review.
  2. Scoring is rubric-based. Two objects with identical scores may take very different times to rewrite in practice.
  3. Business logic complexity is not captured — the tool only analyzes syntax, not semantics.
  4. Rewrite suggestions (if shown) are illustrative only and are NOT production-ready.
  5. Checklist answers are self-reported, not independently verified by the tool.
  6. The checklist is not exhaustive. Real engagements will surface items the tool did not anticipate. Document those and send them to the maintainer so the tool improves.
  7. Scoring weights are opinionated. Other engagements may warrant different weights.
  8. The effort formula is an order-of-magnitude estimate based on code size and complexity. It will vary significantly based on team skill, tooling maturity, and scope.
  9. No data leaves your browser. Nothing is uploaded, logged, or stored.

How confidence is computed

Confidence is computed dynamically from the data you provide — not a fixed number. It starts at a cautious 55% baseline with nothing filled in, and moves from there:

  • +20% when an analyzer run has happened
  • +5% if the analyzed SQL actually contains Sybase-specific patterns (a dialect sanity check)
  • -15% if SQL was analyzed but no Sybase patterns were found — usually means wrong dialect pasted in
  • Up to +20% scaled by the fraction of checklist items answered
  • Floor: 40%. Ceiling: 95%. Nothing in this tool is ever "100% confident".

Confidence also drops sharply (even when the number looks fine) whenever the platform uses features outside the 60+ rule set — custom extended stored procedures, Sybase IQ columnar analytics, Replication Server topology, undocumented thick-client dependencies — or when checklist items are self-reported rather than verified.

What it is NOT

  • Not a SQL parser
  • Not an on-site audit
  • Not a substitute for engaging the DBA or reading the actual code
  • Not a production-ready PostgreSQL rewrite engine
  • Not tuned for non-Sybase rule packs yet (Oracle / SQL Server / MySQL — coming)
